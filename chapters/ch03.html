Convolution: Output Size & Parameter Count

A conv layer has hyperparameters kernel $k$, stride $s$, padding $p$. Output spatial size: $H_{out}=\big\lfloor\frac{H_{in}+2p-k}{s}\big\rfloor+1$. Each filter has shape $(c_{in},k_h,k_w)$ and there are $c_{out}$ filters, so parameters $= c_{out}\,(c_{in}\,k_h\,k_w + 1)$ (the $+1$ is the bias per filter). Common trios: $(k{=}3,s{=}1,p{=}1)$ keeps size; $(k{=}3,s{=}2,p{=}1)$ halves it.

Open🔥 5×SS24 2.8 SS22 1.3 SS21 2.6 SS23 6.1
Parameter-count drills (always include bias): (a) input $10\times20\times30\times40$ (N×C×H×W), single $3\times3$ filter, stride 2 — how many parameters? (b) input $16\times32\times64\times64$, single $1\times1$ filter — how many? (c) first conv on an RGB input, kernel $3\times3$, $16$ output channels?
Answer

Parameters depend only on $c_{in}, k, c_{out}$ — not on spatial size or stride.
(a) $3\times3\times20 + 1 = \mathbf{181}$ (SS24 2.8).
(b) $1\times1\times32 + 1 = \mathbf{33}$ (SS22 1.3).
(c) $(3\times3)\times3\times16 + 16 = 432+16 = \mathbf{448}$ (SS23 6.1).

💡 Extended memory

Formula: $c_{out}(c_{in}k_h k_w + 1)$. Stride/padding/input-size change the *output dimensions*, never the parameter count. Don't forget one bias per output channel.

Open🔥 3×SS20 P4b SS20 P4c
(a) Conv with 5 filters, size 7, stride 3, padding 1 on input $26\times26\times26$ — output dimensionality? (b) Conv with 4 filters, kernel 5, stride 1, no padding on an RGB image — shape of the weight tensor and its dimensions?
Answer

(a) $H_{out}=\lfloor\frac{26+2\cdot1-7}{3}\rfloor+1 = \lfloor\frac{21}{3}\rfloor+1 = 7+1 = 8$ → output $\mathbf{8\times8\times5}$ (5 output channels).
(b) Weight tensor shape $(4,3,5,5)$ = (output channels, input/RGB channels, height, width).

💡 Extended memory

Output-size formula $\lfloor(H+2p-k)/s\rfloor+1$. Weight tensor = (c_out, c_in, k_h, k_w) — exactly 4 dimensions; naming all four earns the marks.

Multiple Choice2×SS24 1.1 SS24 1.7
Which statements about a convolution layer's output are true?

Select all options you think are correct, then check.

Answer

True: output size depends on stride; padding can preserve spatial size. False: receptive fields *grow* with depth (not equal across layers), and parameter count is independent of spatial size.

💡 Extended memory

Two independent facts the exam mixes: *output dimensions* depend on $k,s,p,$ input size; *parameter count* depends only on $c_{in},k,c_{out}$.

AI-generatedpracticeAI-generated
An input $8\times3\times32\times32$ passes through Conv($k{=}3,s{=}1,p{=}1$, 64 filters). Give the output shape and the parameter count.
Answer

Output spatial: $\lfloor(32+2-3)/1\rfloor+1 = 32$, with 64 channels → output $\mathbf{8\times64\times32\times32}$. Parameters: $64\times(3\times3\times3 + 1) = 64\times28 = \mathbf{1792}$.

💡 Extended memory

The $(k{=}3,s{=}1,p{=}1)$ 'magic trio' keeps $H,W$ unchanged — memorize it; it's everywhere in ResNet/VGG.

Why Convolutions (vs FC) & Equivariance

Convolutions beat FC layers for vision because they: extract local features, accept variable input sizes, are parameter-efficient, and are translation-equivariant (shift the input → output shifts the same way). They remain a linear operation. Note: not rotation-invariant in general.

Open2×SS22 5.10
Why do convolutions fit semantic segmentation better than fully-connected layers?
Answer

FC layers process global information from the entire image and require a fixed input size, whereas convolutions operate on local neighbourhoods, are translation-equivariant, and accept variable sizes — exactly what a dense, per-pixel task like segmentation needs. Convolutions also use far fewer parameters.

💡 Extended memory

Segmentation needs spatially-localized, size-flexible predictions → fully-convolutional. FC would destroy spatial structure and explode in parameters.

Open2×Summary III.5 Q6 Summary III.5 Q4
State two differences between a convolutional layer and a pooling layer. Does reducing spatial size reduce the parameters of downstream conv layers?
Answer

Differences: (1) a conv layer has learnable parameters, a pooling layer has none; (2) pooling works channel-wise (kernel depth 1, each channel independently), while a default conv filter spans the full input depth.
Spatial size vs conv params: No — as long as the spatial size stays larger than the kernel, conv parameters depend on $c_{in},k,c_{out}$ only, so they don't change.

💡 Extended memory

Reducing spatial size cuts *computation* and shrinks downstream FC parameters (fixed channel count), but leaves conv parameter counts unchanged.

Open2×Summary III.5 Q7 SS21 1.3
For a fully-convolutional network: can you feed images at double the training resolution? Will performance be the same?
Answer

Yes, fully-convolutional networks accept varying input sizes with no architectural change (the parameter count stays the same). But performance will not be the same — convolutions are not scale-invariant, so objects at a different scale than training won't be recognized as well.

💡 Extended memory

Size-flexible ≠ scale-invariant. Doubling resolution keeps the math valid but changes the effective object scale relative to the receptive field.

AI-generatedpracticeAI-generated
Explain the difference between translation *invariance* and translation *equivariance*, and which one a convolution provides.
Answer

Invariance: the input changes but the output stays the same. Equivariance: the input changes and the output changes the *same way*. A convolution is translation-equivariant — shift the input and the feature map shifts identically. (Global invariance is approached later via pooling and deep receptive fields.)

💡 Extended memory

Conv = equivariant; pooling adds a degree of local invariance. Neither gives rotation invariance by default.

Receptive Field

The receptive field is the region of the input that influences one output feature. It grows with depth, larger kernels, and stride. For a chain, $r_{out}=r_{in}+(k-1)\prod(\text{previous strides})$. Three stacked $3\times3$ (stride 1) convs give a $7\times7$ receptive field. Too-small a receptive field → the output can't see enough context.

Open🔥 3×SS24 2.2 SS20 P4d
Explain the concept of receptive field. What happens if it is too small? (e.g. 3 consecutive $3\times3$ stride-1 convs — how big is the receptive field?)
Answer

The receptive field is the region in the input that affects a particular output feature. If it is too small, the output cannot capture information from the whole input, losing global context and hurting performance.
Example (SS20 P4d): $1\times1\to3\times3\to5\times5\to7\times7$ — three $3\times3$ stride-1 convs give a $\mathbf{7\times7}$ receptive field.

💡 Extended memory

Each $3\times3$ stride-1 conv adds $2$ to the receptive-field width. Stacking small kernels is the cheap way to grow it (VGG's insight).

Open2×SS23 6.2 SS21 2.7
Compute the receptive field of a neuron after several conv/maxpool layers (e.g. Conv3×3 → MaxPool2×2(s2) → Conv3×3 → MaxPool2×2(s2) → Conv3×3).
Answer

Use $r_{out}=r_{in}+(k-1)\cdot\prod(\text{strides so far})$, stepping forward:
Conv1: $3$ → MaxPool1: $3+1\cdot(2-1)=4$ → Conv2: $4+2\cdot(3-1)=8$ → MaxPool2: $8+2\cdot(2-1)=10$ → Conv3: $10+4\cdot(3-1)=\mathbf{18}$ → receptive field $18\times18$ (SS23 6.2).

💡 Extended memory

Track the running product of strides — each later layer's $(k-1)$ contribution is multiplied by all earlier strides, so pooling (stride 2) accelerates receptive-field growth.

Open1×WS26 P5a WS26 P5c
Name an architecture change that exactly doubles the receptive field of a conv layer without adding layers.
Answer

Increase the stride of an earlier layer. E.g. changing Layer 1's stride from 1 to 3 gives receptive field $(6-1)\times3 + 5 = 20$ — doubling it (WS26 P4c). (Alternatively increase an intermediate pooling stride, or enlarge a kernel / use dilation.)

💡 Extended memory

Larger stride or dilation grows the receptive field cheaply but reduces spatial resolution — a trade-off. Pooling layers also increase deeper layers' receptive fields (WS26 1.x).

Pooling & Backprop through Max-Pool

Pooling downsamples channel-wise with no learnable parameters. MaxPool keeps the max (non-linear, sparse gradient); AvgPool keeps the mean (linear, every entry gets gradient). To backprop through MaxPool you must remember the argmax position from the forward pass; only that entry gets a gradient (value 1), the rest get 0.

Open2×Summary III.5 Q9
MaxPool with $k{=}2,s{=}2,p{=}0$, one unique max per window: what fraction of pixels get a live gradient, and what is its value?
Answer

Each $2\times2$ window has 4 entries, only the max gets a gradient → $\frac{1}{4}$ of all entries have a live gradient, and that gradient's value is $1$ (the local derivative of the max w.r.t. its argmax input).

💡 Extended memory

MaxPool routes the upstream gradient entirely to the argmax position. That's why you cache positions in the forward pass — same idea as ReLU caching signs.

Multiple Choice2×SS22 1.7
Given a Max-Pool layer ($2\times2$) on a window with all unique positive values:

Select all options you think are correct, then check.

Answer

True: you need the argmax positions to backprop, and MaxPool performs a form of feature selection (keeps the strongest activation). False: only 25% (1 of 4) of derivatives are non-zero, and MaxPool has no weights to update.

💡 Extended memory

MaxPool: no parameters, non-linear, sparse gradient (1/k² live), selects dominant features. AvgPool would give every entry a gradient.

Open2×Summary V.11 Q1a Summary III.5 Q8
Give one advantage and one disadvantage of average pooling over max pooling. Can we pool without reducing spatial size?
Answer

Advantage of AvgPool: every entry of the window receives a gradient (vs only the max). Disadvantage: average pooling is a linear operation, while max pooling adds non-linearity.
No size reduction: yes — use a size-preserving trio like $(k{=}3,s{=}1,p{=}1)$, just like with convolutions.

💡 Extended memory

Pooling reduces size only because of stride. With stride 1 + matching padding it preserves resolution while still aggregating locally.

1×1 / Special Convolutions

A 1×1 (pointwise) convolution mixes channels at each pixel: it changes the channel dimension cheaply (used for bottlenecks, e.g. Inception) while keeping spatial size. Weight shape $(c_{out}, c_{in}, 1, 1)$. Other special types: depth-wise (per-channel), transpose/up-convolutions (upsampling).

Open🔥 3×SS24 2.1 Summary III.5 Q10 Summary III.5 Q2
Explain the role of 1×1 convolutions. Advantage/disadvantage vs 3×3. For a 1×1 conv with 10 input → 5 output channels, give the weight shape and parameter count.
Answer

Role: mix/reduce the number of channels at each pixel (cheap dimensionality reduction) while preserving spatial size.
vs 3×3: advantage — far cheaper in parameters/compute; disadvantage — a smaller receptive field (no spatial context).
Shape/params: weights $5\times10\times1\times1$, bias $1\times5$ → $5\times10 + 5 = \mathbf{55}$ parameters.

💡 Extended memory

1×1 conv = a per-pixel fully-connected layer across channels. Inception uses it to cut channels before expensive 3×3/5×5 convs.

Open1×Summary III.5 Q1
How can we represent an FC layer with 5 output neurons as a convolutional layer over a $4\times3\times8\times8$ batch? And with a 1×1 convolution?
Answer

An FC layer uses all input neurons, so the equivalent conv needs a kernel matching the full input: $k_h{=}8, k_w{=}8, k_c{=}3$. Weight tensor $5\times3\times8\times8$ (filters × $c_{in}$ × $k_h$ × $k_w$), output $4\times5\times1\times1$. With a 1×1 conv, first reshape the input to $4\times(3\cdot8\cdot8)\times1\times1$ and convolve.

💡 Extended memory

FC ⇔ conv with a full-image kernel. This equivalence is why fully-convolutional nets can replace FC classifier heads.

AI-generatedpracticeAI-generated
In an Inception module, why place a 1×1 convolution *before* a 5×5 convolution?
Answer

The 1×1 conv first reduces the channel count, so the following 5×5 conv operates on far fewer input channels — drastically cutting its parameters and FLOPs (a 'bottleneck') while keeping spatial size. This made GoogLeNet deep yet efficient.

💡 Extended memory

1×1 bottlenecks decouple channel-mixing (cheap) from spatial filtering (expensive). Same trick in ResNet bottleneck blocks.

Handcrafted Kernels & Feature-Map Tokenization

We don't use fixed handcrafted kernels (Sobel, Gaussian) as layers because they are not learnable and can't adapt to the task — though they can serve as initialization or in augmentation. A feature map can be reshaped into tokens for a transformer.

Open2×Summary III.5 Q11 Summary III.5 Q12
Why don't we use handcrafted kernels (e.g. Sobel edge detector) inside deep models? Where can handcrafted kernels (e.g. Gaussian blur) still be used?
Answer

Because handcrafted kernels are not learnable and thus can't adapt to the task — the whole point of deep learning is to *learn* the filters. They can still be used as the initial weights of conv layers (then fine-tuned), or in data augmentation / preprocessing (e.g. Gaussian blur).

💡 Extended memory

Learnable filters > fixed filters because they specialize to the data. Classical kernels survive only as priors/initialization or as augmentation operations.

Open1×Summary III.5 Q13
To feed an intermediate feature map (output tensor of a conv layer) into a transformer to learn pixel relations, how do you reshape the tensor?
Answer

Move the channels to the last dimension and flatten the spatial dims: $(B\times C\times H\times W)\to(B\times H\times W\times C)\to(B\times (H\cdot W)\times C)$. Now each pixel — a $C$-dim vector across channels — is a token for the transformer.

💡 Extended memory

This 'patch/pixel → token' reshape is the bridge from CNN feature maps to attention (the core idea behind Vision Transformers).
