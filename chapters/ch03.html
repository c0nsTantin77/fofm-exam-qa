3.1 Convolution: Output Size & Parameter Count

A conv layer has hyperparameters kernel $k$, stride $s$, padding $p$. Output spatial size: $H_{out}=\big\lfloor\frac{H_{in}+2p-k}{s}\big\rfloor+1$. Each filter has shape $(c_{in},k_h,k_w)$ and there are $c_{out}$ filters, so parameters $= c_{out}\,(c_{in}\,k_h\,k_w + 1)$ (the $+1$ is the bias per filter). Common trios: $(k{=}3,s{=}1,p{=}1)$ keeps size; $(k{=}3,s{=}2,p{=}1)$ halves it.

3.1.1Open🔥 5×SS24 2.8 SS22 1.3 SS21 2.6 SS23 6.1
Parameter-count drills (always include bias): (a) input $10\times20\times30\times40$ (N×C×H×W), single $3\times3$ filter, stride 2 — how many parameters?
(b) input $16\times32\times64\times64$, single $1\times1$ filter — how many?
(c) first conv on an RGB input, kernel $3\times3$, $16$ output channels?
Answer

Parameters depend only on $c_{in}, k, c_{out}$ — not on spatial size or stride.
(a) $3\times3\times20 + 1 = \mathbf{181}$ (SS24 2.8).
(b) $1\times1\times32 + 1 = \mathbf{33}$ (SS22 1.3).
(c) $(3\times3)\times3\times16 + 16 = 432+16 = \mathbf{448}$ (SS23 6.1).

💡 Extended memory

Formula: $c_{out}(c_{in}k_h k_w + 1)$. Stride/padding/input-size change the *output dimensions*, never the parameter count. Don't forget one bias per output channel.

3.1.8Open1×WS24 2.9
Write down the conv weights (one output channel, RGB input, $3\times3$, padding 1, stride 1) so the layer maps an RGB image to grayscale by $Y=0.30R+0.59G+0.11B$.
Answer

Three $3\times3$ kernels (one per input channel), each all zeros except the center: R-kernel center $=0.30$, G-kernel center $=0.59$, B-kernel center $=0.11$. With padding 1 / stride 1, the center-only weight performs a per-pixel weighted sum of the three channels → grayscale.

3.1.9Open1×WS24 3.4
Explain a pro (1p) and a con (1p) of using a larger kernel size in the convolutional layers.
Answer

Pro: a larger kernel gives a larger receptive field per layer, so the model can detect broader patterns earlier.

3.2 Why Convolutions (vs FC) & Equivariance

Convolutions beat FC layers for vision because they: extract local features, accept variable input sizes, are parameter-efficient, and are translation-equivariant (shift the input → output shifts the same way). They remain a linear operation. Note: not rotation-invariant in general.

3.2.1Open2×SS22 5.10
Why do convolutions fit semantic segmentation better than fully-connected layers?
Answer

FC layers process global information from the entire image and require a fixed input size, whereas convolutions operate on local neighbourhoods, are translation-equivariant, and accept variable sizes — exactly what a dense, per-pixel task like segmentation needs. Convolutions also use far fewer parameters.

💡 Extended memory

Segmentation needs spatially-localized, size-flexible predictions → fully-convolutional. FC would destroy spatial structure and explode in parameters.

State two differences between a convolutional layer and a pooling layer. Does reducing spatial size reduce the parameters of downstream conv layers?
Answer

Differences: (1) a conv layer has learnable parameters, a pooling layer has none; (2) pooling works channel-wise (kernel depth 1, each channel independently), while a default conv filter spans the full input depth.
Spatial size vs conv params: No — as long as the spatial size stays larger than the kernel, conv parameters depend on $c_{in},k,c_{out}$ only, so they don't change.

💡 Extended memory

Reducing spatial size cuts *computation* and shrinks downstream FC parameters (fixed channel count), but leaves conv parameter counts unchanged.

3.3 Receptive Field

The receptive field is the region of the input that influences one output feature. It grows with depth, larger kernels, and stride. For a chain, $r_{out}=r_{in}+(k-1)\prod(\text{previous strides})$. Three stacked $3\times3$ (stride 1) convs give a $7\times7$ receptive field. Too-small a receptive field → the output can't see enough context.

3.3.1Open🔥 3×SS24 2.2 SS20 P4d
Explain the concept of receptive field. What happens if it is too small? (e.g. 3 consecutive $3\times3$ stride-1 convs — how big is the receptive field?)
Answer

The receptive field is the region in the input that affects a particular output feature. If it is too small, the output cannot capture information from the whole input, losing global context and hurting performance.
Example (SS20 P4d): $1\times1\to3\times3\to5\times5\to7\times7$ — three $3\times3$ stride-1 convs give a $\mathbf{7\times7}$ receptive field.

💡 Extended memory

Each $3\times3$ stride-1 conv adds $2$ to the receptive-field width. Stacking small kernels is the cheap way to grow it (VGG's insight).

3.3.2Open2×SS23 6.2 SS21 2.7
Compute the receptive field of a neuron after several conv/maxpool layers (e.g. Conv3×3 → MaxPool2×2(s2) → Conv3×3 → MaxPool2×2(s2) → Conv3×3).
Answer

Use $r_{out}=r_{in}+(k-1)\cdot\prod(\text{strides so far})$, stepping forward:
Conv1: $3$ → MaxPool1: $3+1\cdot(2-1)=4$ → Conv2: $4+2\cdot(3-1)=8$ → MaxPool2: $8+2\cdot(2-1)=10$ → Conv3: $10+4\cdot(3-1)=\mathbf{18}$ → receptive field $18\times18$ (SS23 6.2).

💡 Extended memory

Track the running product of strides — each later layer's $(k-1)$ contribution is multiplied by all earlier strides, so pooling (stride 2) accelerates receptive-field growth.

3.4 Pooling & Backprop through Max-Pool

Pooling downsamples channel-wise with no learnable parameters.
MaxPool keeps the max (non-linear, sparse gradient);
AvgPool keeps the mean (linear, every entry gets gradient). To backprop through MaxPool you must remember the argmax position from the forward pass; only that entry gets a gradient (value 1), the rest get 0.

3.4.1Open2×Summary III.5 Q9
MaxPool with $k{=}2,s{=}2,p{=}0$, one unique max per window: what fraction of pixels get a live gradient, and what is its value?
Answer

Each $2\times2$ window has 4 entries, only the max gets a gradient → $\frac{1}{4}$ of all entries have a live gradient, and that gradient's value is $1$ (the local derivative of the max w.r.t. its argmax input).

💡 Extended memory

MaxPool routes the upstream gradient entirely to the argmax position. That's why you cache positions in the forward pass — same idea as ReLU caching signs.

3.5 1×1 / Special Convolutions

A 1×1 (pointwise) convolution mixes channels at each pixel: it changes the channel dimension cheaply (used for bottlenecks, e.g. Inception) while keeping spatial size. Weight shape $(c_{out}, c_{in}, 1, 1)$. Other special types: depth-wise (per-channel), transpose/up-convolutions (upsampling).

3.5.1Open🔥 3×SS24 2.1 Summary III.5 Q10 Summary III.5 Q2
Explain the role of 1×1 convolutions. Advantage/disadvantage vs 3×3. For a 1×1 conv with 10 input → 5 output channels, give the weight shape and parameter count.
Answer

Role: mix/reduce the number of channels at each pixel (cheap dimensionality reduction) while preserving spatial size.
vs 3×3: advantage — far cheaper in parameters/compute; disadvantage — a smaller receptive field (no spatial context).
Shape/params: weights $5\times10\times1\times1$, bias $1\times5$ → $5\times10 + 5 = \mathbf{55}$ parameters.

💡 Extended memory

1×1 conv = a per-pixel fully-connected layer across channels. Inception uses it to cut channels before expensive 3×3/5×5 convs.

Yes. With fixed sizes, a convolution is a linear map with a specific (sparse, weight-shared) structure. A Linear layer can replicate it exactly by setting its weight matrix to match the conv's connectivity and shared weights (zeros where the conv doesn't connect), then reshaping back.

3.6 Handcrafted Kernels & Feature-Map Tokenization

We don't use fixed handcrafted kernels (Sobel, Gaussian) as layers because they are not learnable and can't adapt to the task — though they can serve as initialization or in augmentation. A feature map can be reshaped into tokens for a transformer.

3.6.1Open2×Summary III.5 Q11 Summary III.5 Q12
Why don't we use handcrafted kernels (e.g. Sobel edge detector) inside deep models? Where can handcrafted kernels (e.g. Gaussian blur) still be used?
Answer

Because handcrafted kernels are not learnable and thus can't adapt to the task — the whole point of deep learning is to *learn* the filters. They can still be used as the initial weights of conv layers (then fine-tuned), or in data augmentation / preprocessing (e.g. Gaussian blur).

💡 Extended memory

Learnable filters > fixed filters because they specialize to the data. Classical kernels survive only as priors/initialization or as augmentation operations.

To feed an intermediate feature map (output tensor of a conv layer) into a transformer to learn pixel relations, how do you reshape the tensor?
Answer

Move the channels to the last dimension and flatten the spatial dims: $(B\times C\times H\times W)\to(B\times H\times W\times C)\to(B\times (H\cdot W)\times C)$. Now each pixel — a $C$-dim vector across channels — is a token for the transformer.
